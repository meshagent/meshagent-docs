title: "CLI Agents & the Room Proxy"
---

Terminal-based coding agents such as [OpenAI Codex CLI](https://github.com/openai/codex) and [Claude Code](https://claude.ai/code) use LLMs to reason about code, generate changes, and complete development tasks from the terminal. By default, these tools send model requests directly to a provider such as OpenAI or Anthropic.

The MeshAgent Room Proxy allows you to send model requests to a room-scoped MeshAgent endpoint. MeshAgent then forwards those requests to the upstream provider configured for the room and records usage for the room, participant, and session.

This is useful for individuals who want unified visibility into coding-agent usage, and for organizations that want to manage provider access centrally instead of requiring each developer machine or CI job to authenticate directly with the model provider.

## Why this matters

**For individuals:** You can run multiple coding agents across different projects and see all your LLM usage — costs, models, token counts — in one place attributed to the room and session that produced them.

**For organizations:** Provider credentials can be managed centrally in MeshAgent and configured once per project, instead of being distributed to developer workstations or CI containers. Coding-agent traffic routes through a room-scoped gateway where access, usage, and cost can be monitored and controlled. Developers authenticate with a room-scoped token rather than a raw provider key.

## How it works

The Room Proxy exposes OpenAI- and Anthropic-compatible endpoints at room-scoped URLs:

```
# OpenAI-compatible (Codex CLI, etc.)
https://api.meshagent.com/rooms/{room_name}/openai/v1

# Anthropic-compatible (Claude Code, etc.)
https://api.meshagent.com/rooms/{room_name}/anthropic
```

Instead of authenticating directly with the provider, the CLI authenticates to MeshAgent using a room-scoped participant token. The proxy validates the token, injects the real provider key server-side, forwards the request, and records usage. The coding agent itself is unchanged — it still drives the task, decides when to call the model, and interprets responses. MeshAgent acts purely as the LLM gateway.

## Setup: Shared first steps

These steps apply regardless of which CLI agent you are using.

### 1. Authenticate to MeshAgent and create an API key

If you haven't set up MeshAgent yet, start with `meshagent setup`. Then create an API key. **Copy the printed value — it is shown only once.**

```bash
meshagent api-key create --activate cli-agent-key
```

### 2. Create a token spec and generate a participant token JWT

The Room Proxy requires a **participant token** (JWT) as the bearer credential. The API key from Step 1 is used to sign it. The JWT is scoped to a specific room.

Create a token spec file (contains no secrets — safe to commit):

```yaml token-spec.yaml
version: v1
kind: ParticipantToken
room: my-room
identity: cli-agent
role: agent
api:
  agents: {}
  messaging: {}
  queues: {}
  storage: {}
  memory: {}
  sync: {}
  database: {}
```

Replace `my-room` with your room name, then generate the JWT:

```bash
meshagent token generate --input token-spec.yaml
```

This prints a `eyJ...` JWT. Store it somewhere secure — you will use it as the API key value in your CLI agent's configuration.

> **Token expiration:** Tokens do not expire by default. Regenerate at any time by re-running this command with the same spec file.

### 3. What is the `Meshagent-Session` header?

The proxy requires a `Meshagent-Session` header on every request. It is used purely as a **usage tracking label** — it gets stored alongside each LLM usage record so you can identify which agent session produced which calls in the MeshAgent Studio dashboards. There is no validation against an active room connection. Any non-empty string works; a descriptive name like `"codex-agent"` is fine.

## Codex CLI

[Codex CLI](https://developers.openai.com/codex/cli) supports custom model providers in `~/.codex/config.toml`, including custom base URLs, API key env vars, and static HTTP headers — making it a natural fit for the Room Proxy.

### Install Codex

<CodeGroup>
```bash macOS
brew install openai-codex
```
```bash npm
npm install -g @openai/codex
```
</CodeGroup>

### Store the JWT as an environment variable

Add the JWT from the shared setup to your shell environment:

```bash
export MESHAGENT_ROOM_TOKEN="eyJ..."   # your full JWT here
```

For a persistent setup, add this to a `.env` file that is sourced in your shell profile.

### Configure `~/.codex/config.toml`

Add a MeshAgent provider and profile:

```toml ~/.codex/config.toml
[model_providers.meshagent]
name = "MeshAgent"
base_url = "https://api.meshagent.com/rooms/my-room/openai/v1"
env_key = "MESHAGENT_ROOM_TOKEN"
http_headers = { "Meshagent-Session" = "codex-agent" }

[profiles.meshagent]
model_provider = "meshagent"
model = "gpt-5.4"
```

| Key | Description |
|-----|-------------|
| `base_url` | Replace `my-room` with your room name. |
| `env_key` | The env var name holding the JWT from the shared setup. |
| `http_headers` | Sets `Meshagent-Session` for usage attribution. Any non-empty string works. |
| `model` | Any model available through the OpenAI API. |

### Run Codex with the MeshAgent profile

```bash
codex --profile meshagent "your task here"
```

LLM usage will appear in **MeshAgent Studio → Usage**, attributed to the room and session you configured.

## Claude Code

[Claude Code](https://code.claude.com/docs/en/overview) is Anthropic's CLI coding agent. It uses the Anthropic API and can be pointed at a custom base URL via environment variables.

### Install Claude Code

<CodeGroup>
```bash macOS, Linux, WSL
curl -fsSL https://claude.ai/install.sh | bash
```
```bash Homebrew
brew install --cask claude-code
```
```bash WinGet
winget install Anthropic.ClaudeCode
```
</CodeGroup>

### Configure environment variables

Set the following in your shell environment (or `.env` file):

```bash
export ANTHROPIC_BASE_URL="https://api.meshagent.com/rooms/my-room/anthropic"
export ANTHROPIC_AUTH_TOKEN="eyJ..."   # your JWT from the shared setup
export ANTHROPIC_CUSTOM_HEADERS=$'Meshagent-Session: name-for-your-session'
```

>**Note**: You may want to export the ANTHROPIC_MODEL you want to use as well for example claude-sonnet-4-6. You can also change the model when claude is running by running `/model claude-sonnet-4-6`.

### Run Claude Code
If you have previously logged into Claude with another account be sure to run `claude /logout` before connecting through MeshAgent. 

```bash
claude
```

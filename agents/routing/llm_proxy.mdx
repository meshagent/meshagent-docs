The MeshAgent LLM Proxy lets OpenAI- and Anthropic-compatible clients send requests through MeshAgent instead of directly to the provider. Use it when you want MeshAgent to manage provider credentials, project-level usage, billing, and budget controls for your team. 

For most direct integrations, use the hosted project route:
- OpenAI-compatible base URL: https://api.meshagent.com/openai/v1
- Anthropic raw HTTP base URL: https://api.meshagent.com/anthropic/v1
- Anthropic SDK base URL: https://api.meshagent.com/anthropic

You authenticate with a MeshAgent OAuth access token and include `Meshagent-Project-Id` on each request so MeshAgent can route and attribute usage to the right project and user. 

## Before you start
By default, MeshAgent provides managed OpenAI and Anthropic access. If you want to bring your own provider credentials instead, configure them per project in [Integrations](../../project_admin/integrations).

Use the hosted MeshAgent proxy for raw HTTP requests, official OpenAI and Anthropic SDKs, and agent frameworks or other SDKs. 

MeshAgent also includes setup flows for tools such as [Codex and Claude](./use_codex_claude). The `meshagent setup` command can configure those tools for you automatically so they use MeshAgent-managed keys and routing. 

If your client can call MeshAgent directly, use the hosted project route on this page. If you need localhost URLs and temporary local credentials for a tool running on your machine, see [Local CLI Proxy](./local_cli_proxy).

To inspect your own hosted proxy usage for the current project, open **LLM Proxy** in [MeshAgent Studio](../../interfaces/meshagent_studio) and use the **My Usage** tab. To manage billing, usage, and project budgets, use [MeshAgent Accounts](../../interfaces/accounts).

## Use the LLM Proxy
To call the hosted project route directly, send a normal OpenAI- or Anthropic-compatible request to MeshAgent with:
- A MeshAgent OAuth access token
- A MeshAgent project ID

### Get an access token and project ID
```bash
meshagent setup 
meshagent project list 
meshagent project activate PROJECT_ID # optional: switch the active project
```

`meshagent setup` signs you in and stores an OAuth session locally. `meshagent project list` shows the projects you can use and their IDs. The current project is marked with `*`. Use that project ID, or run `meshagent project activate PROJECT_ID` first if you want to switch projects.

Then export your token and project ID:
```bash
export MESHAGENT_ACCESS_TOKEN="$(meshagent auth token)"
export MESHAGENT_PROJECT_ID=<project_id from meshagent project list>
```

The signed-in user must have permission to use the LLM proxy for the selected project.

### Make a Raw HTTP Request
#### Send an OpenAI-compatible request

```bash
curl "https://api.meshagent.com/openai/v1/responses" \
  -H "Authorization: Bearer $MESHAGENT_ACCESS_TOKEN" \
  -H "Meshagent-Project-Id: $MESHAGENT_PROJECT_ID" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-5.4",
    "input": "Tell me a fun fact about AI."
  }'
```

#### Anthropic-compatible request

```bash
curl "https://api.meshagent.com/anthropic/v1/messages" \
  -H "Authorization: Bearer $MESHAGENT_ACCESS_TOKEN" \
  -H "Meshagent-Project-Id: $MESHAGENT_PROJECT_ID" \
  -H "anthropic-version: 2023-06-01" \
  -H "content-type: application/json" \
  -d '{
    "model": "claude-sonnet-4-6",
    "max_tokens": 512,
    "messages": [
      {"role": "user", "content": "Tell me a fun fact about AI."}
    ]
  }'
```

For hosted Anthropic-compatible requests, use `Authorization: Bearer ...`. Although the Anthropic router can parse `x-api-key`, the hosted OAuth project-route lookup is wired to the `Authorization` header.

### Use Other Frameworks
Most OpenAI- or Anthropic-compatible frameworks work with the hosted proxy if they let you configure:
- The provider base URL
- The API key or bearer token
- Custom request headers

That covers many OpenAI-based and Anthropic-based frameworks, including frameworks layered on top of the official provider SDKs.

For OpenAI-compatible transports:
- Set the base URL to `https://api.meshagent.com/openai/v1`
- Set the API key to your MeshAgent access token
- Add `Meshagent-Project-Id: <project_id>`

For Anthropic-compatible transports:
- Set the base URL to `https://api.meshagent.com/anthropic` for Anthropic SDKs
- Set the base URL to `https://api.meshagent.com/anthropic/v1` for raw HTTP transports
- Send `Authorization: Bearer <meshagent_access_token>` for hosted project-route auth
- Add `Meshagent-Project-Id: <project_id>`

For hosted Anthropic-compatible requests, do not rely on `x-api-key` alone. The hosted project route looks up OAuth auth from the `Authorization` header.

## Local Tools That Need Provider-Style Configuration
Some local tools only support provider-style settings such as `OPENAI_BASE_URL`, `OPENAI_API_KEY`, `ANTHROPIC_BASE_URL`, or `ANTHROPIC_API_KEY`, but still do not provide a reliable way to send custom headers like `Meshagent-Project-Id`.

For those tools, use [Local CLI Proxy](./local_cli_proxy). It runs a temporary local bridge that tracks token usage and forwards OpenAI- and Anthropic-compatible requests through MeshAgent. The local proxy only works while the `meshagent llm proxy` process is running. When you stop that process, the printed local URLs and keys stop working too.

## Legacy Room Route
Older integrations may use room-scoped proxy routes such as `{room_url}/openai/v1/...` or `{room_url}/anthropic/v1/...`. These routes require you to authenticate with a Participant Token that carries the room grant and LLM API grant. 

For new direct integrations, use the hosted project route instead. The hosted route does not require an active room and uses project-level OAuth authentication. Use room-scoped routes only when requests must be tied to a specific active room session.

## How the Hosted Project Route Works
- Your client sends a normal OpenAI-compatible or Anthropic-compatible request to MeshAgent.
- Your client authenticates with a MeshAgent OAuth access token and sends `Meshagent-Project-Id` to choose the project.
- MeshAgent checks that the user has the `llm_proxy` OAuth scope and permission to use the proxy for that project.
- MeshAgent validates the provider path and model, strips client-side provider auth, injects the project's upstream provider credentials, and forwards the request.
- MeshAgent returns the provider-compatible response and records usage for the project and user.

By default, MeshAgent uses MeshAgent-managed routing for OpenAI and Anthropic. To bring your own provider credentials, configure them per project in MeshAgent Studio under the [Integrations](../../project_admin/integrations) tab.

This is also the direct route used by `meshagent ask` and the [MeshAgent Codex and Claude](./use_codex_claude) integrations.

## Authentication Details
- The default CLI login flow requests the full MeshAgent OAuth scope set, which includes `llm_proxy`.
- If you are using your own OAuth client instead of the CLI login flow, request the `llm_proxy` scope.
- The hosted project route requires `Meshagent-Project-Id` when you authenticate with OAuth.
- The signed-in user must have permission to use the LLM proxy for that project.
- The access token expires. When the CLI has a refresh token, `meshagent auth token` refreshes the access token automatically when it is close to expiry.
- If you paste a token into another tool manually, you need to replace it after it expires. Tools such as Codex and Claude avoid that by calling `meshagent auth token` on demand.

## Supported Provider Paths

MeshAgent proxies both OpenAI and Anthropic endpoints.

### OpenAI-Compatible

- `/v1/chat/completions`
- `/v1/responses`
- `/v1/responses/compact`
- `/v1/responses/input_tokens`
- `/v1/embeddings`
- `/v1/audio/speech`
- `/v1/audio/transcriptions`
- `/v1/audio/translations`
- `/v1/models` and `/v1/models/*`
- `/v1/images/*`
- `/v1/realtime` and `/v1/realtime/*`

Supported OpenAI websocket paths are:

- `/v1/realtime`
- `/v1/responses`

### Anthropic-Compatible

- `/v1/messages`
- `/v1/messages/count_tokens`
- `/v1/messages/batches*`
- `/v1/complete`
- `/v1/models` and `/v1/models/*`

Use this page to choose a deployment path and deploy a service with `meshagent`.

Most teams start with `meshagent process deploy` or `meshagent process spec`. If you already need field-level YAML details, go straight to [Service Manifest Reference](./service_manifest_reference).

## Start here

### 1. Choose deployment scope

| If the service should... | Use |
| --- | --- |
| be available across the whole project | `--global` |
| exist in one room only | `--room myroom` |

`meshagent service create` and `meshagent service create-template` always require one of these flags.

A project service is saved at project scope and becomes available in rooms across the project. A room service is saved to one room and follows that room's session lifecycle.

### 2. Choose how to author the service

| If you need to... | Start with | Use this when |
| --- | --- | --- |
| deploy a standard process-backed agent from CLI flags | `meshagent process deploy` | the built-in flags already describe the agent you want |
| review or customize a generated manifest first | `meshagent process spec` | you want MeshAgent to generate `meshagent.yaml` and then edit it |
| deploy a fixed manifest | `kind: Service` + `meshagent service create` | the configuration is fully known ahead of time |
| deploy a reusable template | `kind: ServiceTemplate` + `meshagent service create-template` | prompts, secrets, or settings should be supplied when the service is created |

### 3. Choose what MeshAgent should run

| Runtime | Field | Use this when |
| --- | --- | --- |
| MeshAgent runs your code in a container | `container:` | you want MeshAgent to start and manage the runtime for you |
| MeshAgent routes to something you already host | `external:` | the service already exists somewhere else |

## Common workflows

### Fastest path: deploy from CLI flags

Use `meshagent process deploy` when the built-in CLI flags already describe the process-backed agent you want.

```bash bash
meshagent process deploy \
  --service-name my-chatbot \
  --agent-name my-chatbot \
  --channel chat \
  --require-web-search \
  --require-storage \
  --room myroom
```

This creates a container-based service using the MeshAgent CLI image. Pass `--room myroom` to save it into one room. Omit `--room` to deploy it at project scope.

Tool flags come in two styles:

- `--require-<tool>` keeps the tool always enabled
- `--<tool>` makes the tool available but toggleable in supported UIs

### Generate `meshagent.yaml`, then review it

Use `meshagent process spec` when you want MeshAgent to generate a manifest and you plan to review or customize it before deployment.

```bash bash
meshagent process spec \
  --service-name my-chatbot \
  --agent-name my-chatbot \
  --channel chat \
  --require-web-search \
  --require-storage > meshagent.yaml

# Deploy into one room
meshagent service create --file meshagent.yaml --room myroom

# Deploy across the whole project
meshagent service create --file meshagent.yaml --global
```

`meshagent service create` and `meshagent service create-template` require a deployment scope. Pass either `--room myroom` or `--global`.

### Deploy a fixed `Service`

Use `kind: Service` when the configuration is fixed at deploy time.

<CodeGroup>
  <ServiceSpecChatbot />
</CodeGroup>

```bash bash
meshagent service validate --file meshagent.yaml
meshagent service create --file meshagent.yaml --room myroom
```

Use `--global` instead of `--room myroom` when the service should be available across the project.

### Deploy a reusable `ServiceTemplate`

Use `kind: ServiceTemplate` when the manifest shape stays the same but one or more values should come from the installer or deployment flow.

<CodeGroup>
  <ServiceTemplateSpecChatbot />
</CodeGroup>

```bash bash
meshagent service validate-template --file meshagent.yaml
meshagent service create-template \
  --file meshagent.yaml \
  --value language=Spanish \
  --room myroom
```

Use `ServiceTemplate` for installer-provided values such as prompts, API keys, addresses, or other per-install settings.

### Connect an existing external service

Use `external:` when MeshAgent should route to an HTTP or MCP service you host elsewhere.

For a concrete example, see [Connect to an external MCP server](./external_mcp_service).

## Manage saved services

```bash bash
# List saved services
meshagent service list
meshagent service list --room myroom

# Update a saved service
meshagent service update --file meshagent.yaml --global --id SERVICE_ID
meshagent service update --file meshagent.yaml --room myroom --id SERVICE_ID

# Update a saved ServiceTemplate
meshagent service update-template --file meshagent.yaml --value key=value --global --id SERVICE_ID
meshagent service update-template --file meshagent.yaml --value key=value --room myroom --id SERVICE_ID

# Delete a saved service
meshagent service delete SERVICE_ID
meshagent service delete SERVICE_ID --room myroom
```

You can also manage saved services from MeshAgent Studio.

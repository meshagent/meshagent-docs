title: Deploy Packaged Services
description: Use meshagent package when your service is described as a Python Package object instead of a handwritten meshagent.yaml or image-first workflow.
---

`meshagent package` is the deployment path for services defined as Python `Package` objects.

Use it when your service configuration, deploy assets, and runtime wiring are already expressed in Python and you want the CLI to build and run or deploy that package directly.

## When to use `meshagent package`

`meshagent package` fits between the other deployment paths:

| Deployment path | Best fit |
| --- | --- |
| `meshagent process` | A process-backed agent you can fully describe with CLI flags |
| `meshagent service` | You want an explicit `meshagent.yaml` manifest |
| `meshagent image` | You are working from an image-first workflow |
| `meshagent package` | Your deployable service is already represented as a Python `Package` object |

## How `meshagent package` works

`meshagent package` treats a Python `Package` object as the source of truth for a room service.

At a high level, the flow is:

1. The CLI loads your Python module.
2. It resolves the exported `Package` object.
3. It collects the package's runtime code and deploy assets.
4. If the package needs a custom image, MeshAgent builds that image for you.
5. MeshAgent then either runs the package in a room or deploys it as a saved room service.

This means you do not need to handwrite a `meshagent.yaml` manifest just to describe the runtime, files, and packaging behavior in a separate format. Those details stay in Python as part of the package definition.

## What the module must export

Point `meshagent package` at a Python file that exports either:

- a `Package` object named `main`
- or a zero-argument callable named `main` that returns a `Package`

Example:

```python
from meshagent.agents import Package

main = Package(name="assistant")
```

## Run the package in a room

Use `run` when you want to build and start the package in a room without creating a saved service.

Both `run` and `deploy` are room-scoped today, so `--room` is required:

```bash bash
meshagent package run ./assistant.py --room quickstart
```

The CLI streams the container logs while it runs. If you interrupt it with `Ctrl+C`, the CLI stops and deletes the temporary container.

By default the CLI looks for an export named `main`. Override that with `--name` when needed:

```bash bash
meshagent package run ./assistant.py --name support_bot --room quickstart
```

Use `--verbose` when you want to see the packaged files before the build starts:

```bash bash
meshagent package run ./assistant.py --room quickstart --verbose
```

## Deploy the package as a service

Use `deploy` when the package should become a saved room service:

```bash bash
meshagent package deploy ./assistant.py --room quickstart
```

You can also set a reusable builder name:

```bash bash
meshagent package deploy \
  ./assistant.py \
  --room quickstart \
  --builder-name python-builder
```

## How it differs from `meshagent service`

With `meshagent package`, the Python `Package` object is the source of truth. The CLI loads the module, resolves the package, collects its deploy assets, and then runs or deploys it for you.

With `meshagent service`, you write or generate the service manifest yourself and treat that YAML as the deployment artifact.

Choose `meshagent package` when your service is a Python project and you want code, assets, and runtime wiring to stay together in one Python-defined package object.

Choose `meshagent service` when you want an explicit YAML deploy spec, need service templating or external services, or want a deployment path that is not tied to Python.

title: Credentials and Secret Scopes
description: Choose the right secret type based on which runtime needs the credential and at what scope.
---

MeshAgent has multiple ways to provide credentials, and they solve different problems.

The first question is whether you are giving something access to MeshAgent itself, or giving it a credential for some external system.

## Tokens vs secrets

MeshAgent tokens and MeshAgent secrets are related, but they are not the same:

- a **token** gives a participant or service permission to call MeshAgent APIs
- a **secret** gives that participant or service an external credential such as an API key, database URL, or OAuth token

For room connections, MeshAgent signs a participant token when a person or service joins the room.

- for a person, that token is shaped by the **room grant**
- for a deployed service, that token is shaped by the service's `token` configuration in the service spec

So if you are deciding what someone can do inside a room, you are deciding room grants or token scopes. If you are deciding how that participant authenticates to an external system, you are choosing a secret type.

The key question for secrets is not just "where should I store this secret?" It is:

- which runtime needs it
- whether it is shared or room-specific
- whether MeshAgent needs it before startup or your code needs it during runtime

If you choose the wrong secret type, the credential may be unavailable where you need it, or it may be scoped much more broadly than necessary.

## Secret types

- **Room secrets** are room-scoped and usually identity-scoped. Use them when a credential belongs to a room participant, a room service identity, or an install in a specific room.
- **Project secrets** are shared runtime credentials for deployed services. Use them when the same service-level credential should be reused across a project.
- **Image pull secrets** are registry credentials MeshAgent uses before the container starts. Use them only for pulling private images.

## Quick guide

| If you need to... | Use this |
| --- | --- |
| store a credential for one room, one installer, or one service identity inside a room | [Room Secrets](../secrets/room_secrets) |
| provide a shared runtime secret to a deployed service | [Project Secrets](../secrets/project_secrets) |
| let MeshAgent authenticate to a private container registry | [Image Pull Secrets](../secrets/image_pull_secrets) |

## Runtime mental model

These secret types line up with different points in the lifecycle:

- **Before startup**: image pull secrets help MeshAgent fetch the container image.
- **At service startup and runtime**: project secrets or room secrets are injected into the service environment.
- **During room interactions**: room secrets can also participate in identity-specific flows such as installer-provided secrets, delegated secrets, and OAuth.

This is why image pull secrets are not interchangeable with project or room secrets. They are consumed by MeshAgent before your service process exists.

## Room secrets vs project secrets

This is the choice people usually need to make.

Use **room secrets** when the credential should vary by room, user, installer, or room service identity. Common examples:

- an installer pastes their own API key in Powerboards
- a room service needs a credential owned by that room
- a user completes an OAuth flow and the resulting token should stay in that room context

Use **project secrets** when the credential is shared infrastructure for deployed services. Common examples:

- a shared backend API token
- a project-wide database connection string
- a Stripe secret or other server-side integration key reused across rooms

## Choose by scope

- If the credential belongs to a room or a participant flow, use [Room Secrets](../secrets/room_secrets).
- If the running service needs a shared credential at runtime, use [Project Secrets](../secrets/project_secrets).
- If MeshAgent only needs credentials to pull an image, use [Image Pull Secrets](../secrets/image_pull_secrets).

OAuth-related secret flows for room identities are covered in [OAuth with Room Secrets](../secrets/oauth_flows).

If you need the room-level CRUD and exchange methods themselves, see the [Secrets API](../room_api/secrets).

If you need to control MeshAgent API access, see [API Scopes](../rest_api/api_scopes) and [Participant Tokens](../rest_api/participant_tokens).
